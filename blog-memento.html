💡 What Is Memento?

Memento is our nightly, offline self-evolution engine for local coding agents. It's built on SkillOpt, a Microsoft Research project (arXiv 2605.23904) that applies machine-learning discipline — scored rollouts, bounded edits, and held-out validation gates — to the text inside your SKILL.md files, without touching any model weights.

Core idea: Your SKILL.md is the "trainable weight" of your AI assistant. Memento is the optimizer that trains it — using your own sessions as training data, and a held-out validation gate to ensure only improvements ship.

Think of it like this: every day you work with Devin, you're generating valuable training signal — recurring tasks, patterns, corrections, successful approaches. Without Memento, all of that signal vanishes. With it, a nightly cycle harvests that experience and uses it to make Devin measurably better at your work.

⚙️ How the Sleep Cycle Works

One "night" runs this pipeline automatically:

🌾
Harvest
Pulls sessions from memory, skills, and transcripts
🔍
Mine
Identifies recurring tasks in your sessions
🔄
Replay
Re-runs tasks offline to generate new signal
🔒
Gate
Only accepts edits that improve held-out score
📋
Stage
Proposes changes for your review before adopting

Data Sources It Harvests

  • Devin transcripts — Native ATIF-v1.7 sessions from ~/.local/share/devin/cli/transcripts/*.json with real user↔agent turns
  • Built-in memory — Memories you save via memento's own memory tools (hybrid search, tiers, lessons) — no external server required
  • SKILL.md files — Your existing skill definitions from .devin/skills/ used as training signal

The Validation Gate — Why It's Safe

Unlike naive self-improvement loops, Memento uses a held-out validation gate. A candidate skill edit is only accepted if it strictly improves performance on tasks your agent hasn't seen during training. This means:

  • Skills can only improve or stay the same — never regress
  • Overfitting to your recent sessions is blocked
  • You always review the staged proposal before it goes live

👥 What Each Role Can Achieve

Memento isn't only for developers. Anyone on the team who does the same kind of task over and over in Devin gets sharper at it, because the engine learns from what you keep, edit, and reject. Here's what it means for three common roles — in plain terms.

📐
Product Owner
Turns fuzzy asks into clear, testable stories
  • Draft acceptance criteria in your team's house format — without re-explaining it every time
  • Stop forgetting the boring-but-critical parts: limits, failure paths, analytics events
  • Slice big epics into stories that follow your "definition of ready"
  • Spend your review time on wording, not on fixing the same gaps in every draft
What it learns from: the parts of each draft you rewrite. Your edits quietly become the rubric that keeps the next draft honest. No style guide to maintain.
🔬
Tester / QA
Decides whether "done" is really done
  • Generate test cases that already include negative, auth, and boundary cases — not just the happy path
  • Reuse your stable data-builders and keywords instead of brittle one-offs
  • Tag tests so they land in the right pipeline stage automatically
  • Carry forward the exact edge cases that broke production before, so they never slip again
What it learns from: the strongest signal on the team — your tests actually run. Pass, fail, or flaky is a hard, checkable result the validation gate trusts directly.
⚙️
Developer
Ships and debugs the code behind the feature
  • Get code that follows your codebase's conventions, not generic best practices
  • Stop seeing suggestions you always reject — the agent learns from your reverts
  • Skip the "re-discover the project structure" step at the start of every session
  • Write shorter prompts: "generate UserService" instead of a paragraph of context
What it learns from: your build and test results (a hard pass/fail) plus the patterns you accept versus revert — so the skill converges on how your team actually writes code.
One brain, shared by all three: the engine doesn't care about job titles — only that a task recurs and has a checkable signal. The harder signals (QA's tests, the developer's builds) even help validate improvements that make the PO's stories better.

🎬 One Feature, Three Roles — A Worked Example

Here's the same machinery up close. A team is building "Instant Payout" — withdraw funds in seconds, capped by a per-day SEPA limit. Three people touch it, and each produces a different correctness signal the validation gate knows how to use.

📐
Product Owner · Priya
"Turn this ticket into proper acceptance criteria"
taskKey: general:implement:acceptance-criteria verifier: judge

❌ Before sleep learns her

  • Generic, happy-path-only criteria
  • Forgets the SEPA cap & the failure path
  • Priya rewrites ~50% of every draft

✅ After a week of nights

  • Stories in her house format, INVEST-shaped
  • Always asks for the limit & rejection path
  • Priya edits ~10% — mostly wording

There's no test suite for a user story, so this lands on the judge branch: a rubric is derived from the task — mined from the parts Priya keeps correcting — and an edit ships only if it scores higher.

{

  "verifier": "judge",
  "success": null,            // deferred to the judge
  "rubric": [
    "States the per-day SEPA limit and over-limit behavior",
    "Covers the rejected-payout path, not just happy path",
    "Lists the analytics events to emit"
  ]
}
🔬
Tester / QA · Tom
"Generate the test cases for this story"
taskKey: python:test:instant-payout verifier: tests

❌ Before

  • Happy-path only; misses boundaries
  • Forgets auth-failure & over-limit cases
  • Brittle selectors that go flaky

✅ After

  • Auto-adds negative, auth & boundary cases
  • Knows the over-limit amount that broke prod
  • Reuses stable data-builder keywords

Tom has the strongest signal on the team: his output runs. Pass, fail, or flaky is a hard, checkable result — no judge needed. An edit is accepted only if its tests actually pass.

✅ After

📊 What Actually Changes After a Few Nights

Beyond the benchmark numbers, here's the progression you can expect as Memento runs on your real sessions.

Week 1 — The agent stops asking obvious questions

Before, every code review session started with the agent re-discovering your project structure and conventions. After the first sleep cycle, it already knows your package layout, your naming patterns, your preferred error handling style — because it mined them from your past sessions and baked them into the skill.

Week 2 — Recurring tasks get shorter to write

Instead of writing "generate a Spring Boot service with repository, DTO, validator, and Mockito tests following our standards", you write "generate UserService". The evolved skill already carries all the context. Your prompts shrink because the skill has absorbed your patterns.

Week 3 — The corrections stop repeating

You used to correct the same things every session — "don't use field injection", "always add @Transactional on service methods", "use ResponseEntity not raw objects". Sleep mines those corrections from your saved memories and encodes them permanently into the skill. You stop seeing the same mistake twice.

Month 2 — A new developer opens the project

They get an AI assistant that already knows your team's conventions — not because someone wrote documentation, but because two months of real sessions were consolidated into the skills. The agent teaches the new developer your standards before the new developer has to ask.

The compounding effect: Each night adds a small improvement. After a week you feel it. After a month it's significant. The more the agent is used, the more material it has to learn from — and the better it gets at the things your team actually does every day.

🛠️ The MCP Tools You Get

Memento is integrated as an MCP server in Devin, giving you 24 tools directly in your Devin chat — the sleep-cycle tools that evolve your skills, plus a full set of built-in memory tools (see Built-in Memory below).

Sleep cycle

memento_auto
Fully automatic. Runs a cycle and auto-adopts the staged proposal when it clears the held-out validation gate, then shows you the SKILL.md diff. Schedulable for hands-off nightly improvement.
memento_status
Check progress. Shows how many nights have run and the latest staged proposal waiting for your review.
memento_dry_run
Preview without committing. Runs the full harvest → mine → replay cycle and reports what it found, without staging anything. Great for understanding what would change.
memento_run
Run a full night. Executes the complete cycle and stages a proposal for your review. Nothing changes in your live skills until you call memento_adopt.
memento_adopt
Apply the improvement. Applies the staged proposal to your SKILL.md and syncs it to your Devin workspace so Devin picks it up immediately.
memento_harvest
Debug / explore. Lists the recurring tasks mined from recent sessions so you can see exactly what the engine is learning from.

Built-in memory (no external server)

memory_save / recall
Remember & retrieve. Save memories (tiered, secret-redacted) and search them with hybrid BM25 + semantic vector retrieval. Saved memories feed the next sleep cycle automatically.
memory_learn / lessons
Distill insights. Mine recurring entities, tags, and failures into semantic-tier lessons — or pin your own.
memory_graph / related
Connect the dots. A knowledge graph of entities extracted from your memories, with neighbour lookup.
memory_dashboard
See it all. Opens a local web dashboard — overview, memory browser, force-directed graph, lessons, sessions, activity.
+ more
consolidate · pin · namespaces · snapshot / restore · audit · capture hooks — agentmemory-class coverage, all owned by memento and stdlib-only.

🗂️ Built-in Memory — With a Dashboard

Memento now ships its own memory engine — no external memory server, no Node. It's stdlib-only (SQLite + a tiny web server): hybrid BM25 + semantic vector search, four memory tiers with auto-consolidation & decay, a knowledge graph, auto-derived lessons, namespaces, an audit log, and snapshots — all browsable in a local web dashboard.

👤 Solo mode (default)

  • Zero config — local SQLite at ~/.memento/memory.db
  • Everything runs on your machine; nothing leaves it

👥 Team mode

  • Set MEMENTO_DB_URL to a shared Postgres (pgvector)
  • Whole team reads/writes one store, scoped per team by namespace

Same tools and the same dashboard in both modes — switching from solo to a shared team brain is a single environment variable.

  • Hybrid retrieval — BM25 full-text fused with semantic vector similarity (Reciprocal Rank Fusion)
  • Tiers + lifecycle — working → episodic → semantic → procedural, with auto-consolidation, decay, and pinning
  • Knowledge graph — entities extracted from memories, with a draggable force-directed view
  • Lessons — recurring entities, tags, and failures are distilled into semantic-tier lessons
  • Governance — namespaces, audit log, and git-versionable snapshot / restore
  • Secret redaction before storage; agentmemory-compatible export so the sleep cycle harvests it automatically
Open the dashboard
python3 mcp_server.py --web --port 3114   # → http://127.0.0.1:3114
Honest scope: embeddings are deterministic term-frequency vectors (real vector cosine, no model needed) — swap in a neural embedder for synonym-level semantics. This is agentmemory-class core coverage (~18 memory tools), not a byte-for-byte clone of its 53-tool surface.
