💡 What Is SkillOpt-Sleep?

SkillOpt-Sleep is a nightly offline self-evolution engine for local coding agents. It's a research project by Microsoft (arXiv 2605.23904) that applies machine-learning discipline — epochs, learning rates, validation gates — to the text inside your SKILL.md files, without touching any model weights.

Core idea: Your SKILL.md is the "trainable weight" of your AI assistant. SkillOpt-Sleep is the optimizer that trains it — using your own sessions as training data, and a held-out validation gate to ensure only improvements ship.

Think of it like this: every day you work with Windsurf or Devin, you're generating valuable training signal — recurring tasks, patterns, corrections, successful approaches. Without SkillOpt-Sleep, all of that signal vanishes. With it, a nightly cycle harvests that experience and uses it to make your agent measurably better.

⚙️ How the Sleep Cycle Works

One "night" runs this pipeline automatically:

🌾
Harvest
Pulls sessions from agentmemory, skills, and logs
🔍
Mine
Identifies recurring tasks in your sessions
🔄
Replay
Re-runs tasks offline to generate new signal
🔒
Gate
Only accepts edits that improve held-out score
📋
Stage
Proposes changes for your review before adopting

Data Sources It Harvests

  • agentmemory — Memories saved by the MCP agentmemory server across all your sessions
  • SKILL.md files — Your existing skill definitions used as training signal
  • Windsurf / Cascade logs — Task snippets extracted from IDE session logs

The Validation Gate — Why It's Safe

Unlike naive self-improvement loops, SkillOpt-Sleep uses a held-out validation gate. A candidate skill edit is only accepted if it strictly improves performance on tasks your agent hasn't seen during training. This means:

  • Skills can only improve or stay the same — never regress
  • Overfitting to your recent sessions is blocked
  • You always review the staged proposal before it goes live

The compounding effect: Each night adds a small improvement. After a week you feel it. After a month it's significant. The more the agent is used, the more material it has to learn from — and the better it gets at the things your team actually does every day.

🛠️ The 5 MCP Tools You Get

SkillOpt-Sleep is integrated as an MCP server in Windsurf and Devin, giving you these tools directly in your AI assistant:

sleep_status
Check progress. Shows how many nights have run and the latest staged proposal waiting for your review.
sleep_dry_run
Preview without committing. Runs the full harvest → mine → replay cycle and reports what it found, without staging anything. Great for understanding what would change.
sleep_run
Run a full night. Executes the complete cycle and stages a proposal for your review. Nothing changes in your live skills until you call sleep_adopt.
sleep_adopt
Apply the improvement. Applies the staged proposal to your SKILL.md and syncs it to your Windsurf workspace so the agent picks it up immediately.
sleep_harvest
Debug / explore. Lists the recurring tasks mined from recent sessions so you can see exactly what the engine is learning from.

🔧 How We Wired It Into Our Devin IDE Setup

Getting SkillOpt-Sleep to work reliably across Devin IDE, Windsurf, Codeium, and IntelliJ took more than just installing an MCP server. Here's the exact setup we built — and the problems we ran into along the way.

The Starting Problem: Config Sprawl

MCP server configurations were scattered across six different files, skills were duplicated in five separate project directories, and SkillOpt-Sleep itself was disconnected from all of them. Changing one server's config meant updating multiple files by hand. Something would always be out of sync.

❌ Before

  • MCP config in 6 different files
  • Skills duplicated across 5 projects
  • skillopt-sleep missing everywhere
  • IntelliJ uses a different XML format
  • Manual sync required across IDEs

✅ After

  • One master mcp_config.json
  • All IDEs symlinked to master
  • Skills in one global location
  • IntelliJ synced via a Python script
  • skillopt-sleep connected and working

Step 1 — One Master Config to Rule Them All

We created a single source of truth at /your/project/mcp_config.json containing all MCP servers. Then we replaced every IDE-specific config with a symlink pointing to it:

bash — symlink all IDE configs to master 
# Windsurf + Devin CLI read from here
ln -sf /your/project/mcp_config.json \
    ~/.codeium/windsurf/mcp_config.json

# Codeium global
ln -sf /your/project/mcp_config.json \
    ~/.codeium/mcp_config.json

# IntelliJ needs XML — generated by sync script
python3 /your/project/mcp-manager/sync-intellij-mcp.py

IntelliJ uses a proprietary XML format (llm.mcpServers.xml) rather than JSON, so we wrote a small Python script that reads the master JSON and generates the correct XML into ~/.config/JetBrains/IntelliJIdea2026.1/options/. Run it once after any change to the master config and IntelliJ picks it up on restart.

Step 2 — One Global Skills Directory

Skills were copy-pasted across every project. A fix in one place didn't propagate anywhere. Windsurf and Devin both read from ~/.codeium/windsurf/skills/ as a global location, so we moved all 15 skills there and replaced every per-project directory with a symlink:

bash — one global skills location 
# Move master skills to global location (do this once)
cp -r /your/project/.windsurf/skills \
    ~/.codeium/windsurf/skills

# Replace every per-project copy with a symlink
GLOBAL=~/.codeium/windsurf/skills

# Repeat for each project you have
rm -rf /your/project-a/.windsurf/skills
ln -s $GLOBAL /your/project-a/.windsurf/skills

rm -rf /your/project-b/.windsurf/skills
ln -s $GLOBAL /your/project-b/.windsurf/skills

# For Devin-only projects (.devin/skills instead of .windsurf/skills)
rm -rf /your/project-c/.devin/skills
ln -s $GLOBAL /your/project-c/.devin/skills

Now editing a skill in ~/.codeium/windsurf/skills/code-review/SKILL.md instantly propagates to every project. SkillOpt-Sleep's sleep_adopt writes there too, so evolved skills are automatically available everywhere.

Step 3 — Fixing the Devin IDE Connection

This was the trickiest part. Even after adding skillopt-sleep to the master config, the Devin IDE kept showing it as disconnected. The MCP log revealed the real story:

~/.config/Devin/logs/.../MCP.log 
# The IDE was loading from a path that no longer existed!
[warning] MCP server 'skillopt-sleep' connection failed: connection closed
[info]    Starting: "python3" ["/home/user/tools/skillopt-windsurf/mcp_server.py"]
#                            ↑ this path did not exist

The Devin IDE stores MCP server configs in its internal LevelDB database, separate from any JSON config file. SkillOpt-Sleep had been manually added through the IDE's UI at some point, pointing to an old path. Changing the JSON config had no effect on it.

Rather than modifying the IDE's internal database (risky), we created a shim script at the exact path the IDE expects. The shim sets the correct environment variables and delegates to the real server:

~/tools/skillopt-windsurf/mcp_server.py 
#!/usr/bin/env python3
# Shim: delegates to the real server with correct env vars.
import os, sys

os.environ.setdefault("SKILLOPT_REPO",    "/path/to/SkillOpt")
os.environ.setdefault("SKILLOPT_PROJECT", "/your/project")

real = "/your/project/.windsurf/skillopt/mcp_server.py"
os.chdir(os.path.dirname(real))
sys.argv[0] = real
exec(compile(open(real).read(), real, "exec"))

The Devin IDE's stored config now resolves correctly, the server initializes, and all five SkillOpt-Sleep tools are available in the IDE's MCP panel.
